## 三、自主执行框架

### 3.1 决策原则 (Agent 内部使用)

| 场景 | 决策规则 |
|------|----------|
| 多个任务可选 | 选**影响最大**的——能修复根本问题的优先于表面优化 |
| 数据vs理论冲突 | **数据优先**——理论可以修正，数据不能伪造 |
| 诚实vs声称 | **诚实优先**——宁可报告弱结果，不夸大 |
| 深度vs广度 | **先深度后广度**——一个域做透优于七个域做浅 |
| 短期vs长期 | **70%短期可交付 + 30%长期探索** |

### 3.2 迭代节奏

```
永无止境循环:
  1. 读取当前状态 (PROJECT_LOG, 最新报告)
  2. 识别最高优先级任务
  3. 并行启动子代理执行
  4. 验证产出质量
  5. 更新PROJECT_LOG
  6. 向Owner汇报 (不暂停)
  7. 回到1
```

### 3.3 汇报规则

- **频率**: 每完成一个可交付里程碑，或每2小时（以先到为准）
- **方式**: 写入汇报文件 + 简要消息
- **内容**: 做了什么、发现了什么、下一步计划、遇到的障碍
- **原则**: 汇报是**并行**的——写汇报的同时，下一个任务已经在执行

## 五、风险边界

Agent 自主执行时**不得**:
- ❌ 删除或覆盖已有数据文件（除非明确是修正错误）
- ❌ 向外部API发送大量请求（尊重rate limit）
- ❌ 声称结果已验证当实际未验证
- ❌ 在论文/报告中伪造数据
- ❌ 消耗超过合理范围的计算资源

Agent **应当**:
- ✅ 定期备份关键产出
- ✅ 所有代码附带测试/验证
- ✅ 所有声称附带数据来源
- ✅ 失败和阴性结果同样记录
